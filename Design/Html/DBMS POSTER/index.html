Normalization in DBMS

A Cheat Sheet for Organizing Your Database Effectively

Why Do We Need Normalization?

Primary Goal

To organize database tables to minimize data redundancy and improve data integrity.

Key Benefits

  • Reduces data repetition.
  • Ensures data is consistent.
  • Makes the database more efficient.

Avoids Anomalies

  • Insertion: Difficulty adding new data.
  • Update: Changes required in multiple places.
  • Deletion: Unintentional loss of data.

The Normal Forms

First Normal Form (1NF)

Rule: Each cell must contain a single, atomic value. No repeating groups or lists.

BAD (Not in 1NF)

StudentIDNameCourses
1AnneMath, Science
2BenHistory

GOOD (In 1NF)

StudentIDNameCourse
1AnneMath
1AnneScience
2BenHistory

Second Normal Form (2NF)

Rule: Must be in 1NF. All non-key attributes must depend on the *entire* primary key (no partial dependencies).

BAD (Not in 2NF)

Primary Key is {StudentID, CourseID}. However, `Teacher` only depends on `CourseID`.

StudentIDCourseIDTeacherGrade
1C101Mr. SmithA
2C101Mr. SmithB

GOOD (In 2NF) - Split into two tables

Enrollment Table

StudentIDCourseIDGrade
1C101A
2C101B

Courses Table

CourseIDTeacher
C101Mr. Smith

Third Normal Form (3NF)

Rule: Must be in 2NF. There should be no transitive dependencies (non-key attributes depending on other non-key attributes).

BAD (Not in 3NF)

`DeptName` depends on `DeptID`, which is not the primary key. This is a transitive dependency.

StudentIDNameDeptIDDeptName
1AnneD1Science
2BenD2Arts

GOOD (In 3NF) - Split into two tables

Students Table

StudentIDNameDeptID
1AnneD1
2BenD2

Departments Table

DeptIDDeptName
D1Science
D2Arts

Boyce-Codd Normal Form (BCNF)

Rule: Must be in 3NF. Every determinant must be a superkey (no overlapping candidate keys issues).

BAD (Not in BCNF) - College Course Enrollment

Functional Dependencies: {StudentID, Subject} ‚Üí Professor, Professor ‚Üí Subject
Candidate Keys: {StudentID, Subject}, {StudentID, Professor}
Problem: "Professor ‚Üí Subject" violates BCNF (Professor is not a superkey)

StudentIDSubjectProfessor
101PhysicsDr. Green
101MusicDr. Blue
102PhysicsDr. Green
103ChemistryDr. White
Anomalies: Update Dr. Green's subject requires multiple changes, can't add professors without students, deleting students loses professor info.

GOOD (In BCNF) - Decomposed Tables

Professor_Subject Table

ProfessorSubject
Dr. GreenPhysics
Dr. BlueMusic
Dr. WhiteChemistry

Student_Professor Table

StudentIDProfessor
101Dr. Green
101Dr. Blue
102Dr. Green
103Dr. White

Fourth Normal Form (4NF)

Rule: Must be in BCNF. No multi-valued dependencies (independent multi-valued facts should be in separate tables).

BAD (Not in 4NF) - Pizza Restaurant Info

Problem: Pizza varieties and delivery areas are independent but stored together, creating unnecessary redundancy.

RestaurantPizzaVarietyDeliveryArea
Pizza HouseMargheritaDowntown
Pizza HouseMargheritaUptown
Pizza HousePepperoniDowntown
Pizza HousePepperoniUptown
Problem: Adding new pizza = N rows (for each area). Adding new area = M rows (for each pizza). Cartesian product explosion! üò©

GOOD (In 4NF) - Separated Independent Facts

Restaurant_Pizzas Table

RestaurantPizzaVariety
Pizza HouseMargherita
Pizza HousePepperoni

Restaurant_Deliveries Table

RestaurantDeliveryArea
Pizza HouseDowntown
Pizza HouseUptown

Fifth Normal Form (5NF)

Rule: Must be in 4NF. Eliminate join dependencies - data must not be reconstructible from smaller tables without loss.

BAD (Not in 5NF) - Sales Agent Authorization

Business Rule: Agent can sell a product only if they work for the company AND are authorized for that product type.

AgentCompanyProduct
Agent ABrandCorpWidget
Agent ABrandCorpGadget
Agent BBrandCorpWidget
Agent BACMEZapper
Problem: Hidden dependencies make it easy to store invalid combinations (e.g., Agent A selling ACME products).

GOOD (In 5NF) - Fundamental Relationships

Agent_Assignments

AgentCompany
Agent ABrandCorp
Agent BBrandCorp
Agent BACME

Product_Lines

CompanyProduct
BrandCorpWidget
BrandCorpGadget
ACMEZapper

Agent_Capabilities

AgentProduct
Agent AWidget
Agent AGadget
Agent BWidget
Agent BZapper
Result: JOIN these three tables to reconstruct valid sales assignments without redundancy or invalid combinations! ‚ú®
